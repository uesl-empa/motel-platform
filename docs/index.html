MOTEL Platform

Methodology for Open Technology Data in Energy Models

MOTEL provides a reproducible workflow for preparing, harmonising, documenting, and reusing technology data in energy system models, while creating a clean handoff into ontology, graph database, API, and web application layers.

Full methodology GitHub repository

What is MOTEL for?

Energy system models rely on many technology assumptions such as CAPEX, efficiency, lifetime, emissions factors, and carrier relations. Those assumptions are often buried in spreadsheets or model-specific files, which makes them hard to trace, compare, and reuse. MOTEL addresses that by introducing a structured workflow for turning raw technology information into harmonised, provenance-aware, and ontology-ready data.

Traceable parameters

Technology values stay connected to source references, assumptions, and harmonisation history.

Reusable structures

Shared schemas, controlled vocabularies, and linked entities make the data easier to compare across studies and tools.

Ontology-aware handoff

Structured MOTEL outputs are mapped in this repository into ontology-ready TTL that can feed downstream graph, API, and web interface layers.

Workflow

The MOTEL workflow is easier to read as two connected stages: the current `motel-platform` ingestion and harmonisation pipeline, then the ontology mapping in this repository followed by the downstream knowledge-graph layer.

  • Traceable technology parameters
  • Clear source documentation
  • Ontology-supported interoperability
  • Reusable data structures
  • Reproducible data preparation workflow
  • Model-ready export as a downstream goal

View Stage 1 source View Stage 2 source

Because the workflow is defined through explicit schemas and controlled vocabularies, it does not depend only on project-specific data transformation scripts. The same structure also makes the process suitable for LLM-assisted acceleration.

In this repository, a local Ollama model (qwen3:14b) is used during harmonisation to help standardise entity names, semantically match staged records to existing registries when exact matches are missing, fill required schema fields for new entries, and create or normalise attribute and scope vocabulary entries.

Methodology

  1. Data ingestion: raw or semi-structured technology data is collected and stored as unmapped records.
  2. Harmonisation: staged records are transformed into a consistent MOTEL data structure.
  3. Entity linking: technologies, parameters, carriers, scopes, and sources are represented through linked records and registries.
  4. Ontology mapping: harmonised MOTEL outputs are converted in this repository into ontology-ready TTL linked to ontology classes and properties.
  5. Data exploration: published MOTEL datasets can be queried and inspected through a notebook-first exploration layer in this repository.
  6. Application and export: the graph database, backend API, and webapp support broader exploration and future model-ready export workflows.

Repository Structure

The current repository focuses on ingestion, harmonisation, ontology mapping, and documentation. A second repository, BartonChenTW/motel-webapp, carries the downstream application stack, including backend, graph-serving, and frontend layers. 

motel-platform/
|-- 1_ingest/         source-specific ingestion notebooks and helpers
|-- 2_harmonise/      harmonisation notebooks and helper functions
|-- 3_ontology_mapping/ ontology-ready TTL generation scripts and outputs
|-- 4_data_explore/   notebook-first exploration of published MOTEL data
|-- motel-db/         unmapped data, registries, mappings, and linked outputs
|-- schema/           machine-readable data schemas
|-- schema_human/     human-readable schema blueprints
`-- docs/             GitHub Pages documentation and workflow figures

Example Technology Record

A harmonised MOTEL record can connect a technology, its parameters, units, source references, and ontology-ready identifiers in one inspectable structure. 

technology:
  id: heat_pump
  label: Heat pump
  type: EnergyConverter
  parameters:
    - id: heat_pump_capex
      label: CAPEX
      value: 1200
      unit: CHF/kW
      source_id: SRC_001
      ontology_class: dici_onto:CAPEX
    - id: heat_pump_lifetime
      label: Lifetime
      value: 25
      unit: years
      source_id: SRC_001
      ontology_class: dici_onto:Lifetime

Ontology and Graph Database Layer

The downstream MOTEL repository BartonChenTW/motel-webapp turns harmonised MOTEL data into ontology-based TTL files, loads them into GraphDB, serves them through a backend API, and exposes them in a web interface for exploration. 

Heat pump
-> hasAttribute
-> CAPEX
-> hasValue
-> 1200 CHF/kW
-> wasDerivedFrom
-> Source record

Webapp and Model Builder

In the separate downstream stack hosted at BartonChenTW/motel-webapp, the MOTEL webapp is intended to let users browse technologies, inspect parameters and provenance, adapt selected values, and export model-ready configuration files.

  • Technology selection
  • Parameter table
  • Source and provenance display
  • Editable or customisable values
  • YAML export

How to Run

This repository can be explored locally with the notebook-based ingestion and harmonisation workflow. The downstream ontology, KG, backend, and frontend stack runs from the separate repository BartonChenTW/motel-webapp. 

git clone https://github.com/BartonChenTW/motel-platform
cd motel-platform
python -m venv .venv
# Windows PowerShell:
# .\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

Then open the workflow notebooks locally in Jupyter or VS Code:

  • 1_ingest/1_data_ingestion.ipynb
  • 2_harmonise/2_data_harmonisation.ipynb
  • 3_ontology_mapping/3_ontology_mapping.ipynb
  • 4_data_explore/4_data_exploration.ipynb

Or run the ontology mapping step directly after harmonisation: 

.\.venv\Scripts\python.exe 3_ontology_mapping\scripts\gen_ttl.py

Outputs and Deliverables

  • A documented methodology for technology data harmonisation
  • A structured technology data model
  • A demonstration technology database
  • Controlled vocabularies, secondary registries, and mapping tables
  • Linked-entity target structures
  • A workflow handoff into ontology, graph, API, and web-app layers
  • Documentation and training material

Status and Limitations

Current Status

  • Data ingestion and harmonisation workflow implemented
  • Unmapped and linked data structures defined
  • Controlled vocabularies and mapping tables generated
  • Ontology-mapping workflow and graph-layer handoff documented
  • Notebook-first exploration entrypoint added for published MOTEL data

Current Limitations

  • The public repo currently demonstrates a selected set of technologies
  • Ontology mapping, GraphDB, backend, and frontend are in the separate repository BartonChenTW/motel-webapp
  • Model-ready YAML export is a downstream target, not a full feature of this repo alone
  • Additional validation and richer export formats remain future work
